# torch2vk Goal

`torch2vk` 的目标不是做一个通用编译器，也不是做一个“什么模型都能接”的推理框架。

它的目标是：为少数已知模型家族提供一个 **Python-first、Vulkan-only** 的推理系统，让人和 AI 都能直接面对可读、可验证、可改写的模型执行逻辑，而不是被导出图、编译器 IR、隐藏 pass 或 runtime 黑盒隔开。

## 它最终是什么

`torch2vk` 最终应该是一套面向少数已知模型家族的 **shader-first 执行系统**。

它不是一个只暴露统一入口、内部执行结构不可见的黑盒 runtime。  
它也不是一个先导出模型、再经过 lowering 和 pass、最后只剩下运行结果的编译管线。

它最终更像是下面这些东西的组合：

1. 一组按模型家族组织的 Python-authored 执行源码和 shader 调用关系。
2. 一组明确、可命名、可替换的 shader variant。
3. 一套围绕真实权重、真实状态和真实张量边界建立起来的运行语义。
4. 一套直接写在模型家族 Python 源码里的多步执行方式，由 eager execution 驱动 Vulkan 执行。
5. 一个小而显式的 Vulkan 执行层。
6. 一套能持续回到 PyTorch reference 做比较和诊断的验证体系。

所以 `torch2vk` 最终不是“一个万能执行引擎”。  
它最终是一套**看得见执行源码、看得见 shader、看得见状态、看得见优化、看得见偏差位置**的系统。

## 它服务什么范围

`torch2vk` 不追求支持任意 Python 模型。

它面向的是：

1. 少数已知的大模型家族。
2. 结构已经比较清楚、值得长期维护的模型体系。
3. 需要显式 shader variant 和显式优化版本的场景。
4. 需要强可见性、强可比较性和强可调试性的推理工作负载。

典型对象包括：

1. Qwen
2. Gemma
3. Llama
4. Whisper 一类结构稳定的语音模型
5. 分段明确的 TTS 系统

这里的关键不是“覆盖尽可能多”，而是“对被选中的模型家族给出真正可控的执行与优化路径”。

## 最终态

`torch2vk` 的最终态应该具备下面这些特征。

### 1. Python 是执行逻辑和 shader 调用关系的真源

在 `torch2vk` 里，重要的执行逻辑应该直接存在于 Python 源码中。

如果一个人想理解某个模型执行片段，或者一个 AI 想改写某个优化版本，它应该能直接通过阅读源码看见：

1. 有哪些张量。
2. 有哪些 op。
3. 执行顺序是什么。
4. 权重来自哪里。
5. 状态存在什么位置。
6. shader 的输入输出边界是什么。

系统不应该要求人先理解某种导出格式、某种 compiler IR、某种 lowering 结果，才能知道模型最后到底在执行什么。

### 2. shader variant 是显式存在的

一个模型执行片段不应该只有“一个内部表示加上一堆隐藏变换”。

在 `torch2vk` 里，不同的 shader variant 应该是显式存在、可以单独命名和单独比较的对象，例如：

1. `linear_bf16_raw`
2. `linear_bf16_tiled`
3. `rotary_embedding`
4. `sdpa_kv_packed`

这些 variant 不是同一份隐藏表示的内部状态，而是系统表面上真实存在的不同 shader 版本。

这意味着系统必须始终能回答：

1. reference 版本是什么。
2. 优化版本是什么。
3. 两者差别在哪里。
4. 哪些差别只是执行形式变化。
5. 哪些差别已经改变了权重格式或状态语义。

### 3. shader 是核心执行单元

系统的核心单位应该是 shader，而不是抽象到失去语义的通用 node 集合。

一个 shader 代表一个清楚、稳定、可命名的计算片段，例如：

1. `rms_norm_bf16`
2. `linear_bf16_raw`
3. `rotary_embedding`
4. `sdpa_attn_weights_softmax`
5. `sdpa_kv_cache_update`

shader 应该足够具体，以至于人或 AI 在读到它时，能直接知道：

1. 它处理哪一类计算语义。
2. 它接收什么输入。
3. 它读写什么张量或状态缓冲。
4. 它产生什么输出。
5. 它和前后 shader 调用的关系是什么。

### 4. 多步执行应直接写在 Python 里

`torch2vk` 面向的不只是单次 forward，而是完整的多步工作负载。

LLM、ASR、TTS 这类系统本质上都不是“一次调用，一个结果”：

1. LLM 有 tokenize、prefill、decode、sample、再 decode。
2. ASR 有特征提取、编码、解码、后处理。
3. TTS 有文本编码、声学解码、vocoder。

`torch2vk` 最终应该把这些流程直接写在各个模型家族自己的 Python 源码里，由 eager execution 按源码顺序驱动 Vulkan shader 执行，而不是再引入一层额外的 workflow 抽象，或者期待某个 runtime 在内部“推断出完整工作流”。

因此系统最终交付的，不只是 kernel，也不只是一份中间表示，而是**直接存在于模型家族 Python 源码中的、完整可阅读的 shader 组合方式**。

### 5. Vulkan 是计算面，不是语义面

`torch2vk` 是 Vulkan-only，不是因为要对齐多个 backend，而是因为它希望计算面足够明确、足够稳定、足够贴近真实执行成本。

在最终系统里，Vulkan 代表的是：

1. 重计算发生的地方。
2. 权重和中间张量真正驻留的地方。
3. 状态持续存在的地方。
4. 性能成本真正发生的地方。

但 Vulkan 不应该成为语义来源。  
模型的意义、执行逻辑的意义、shader 的意义，以及 Python 源码中的多步组合关系，不应该藏在 Vulkan runtime 内部。

## 系统中的几个关键关系

理解 `torch2vk`，最重要的是理解下面这组关系。

### 1. PyTorch eager 是 reference interpreter

PyTorch eager 代表模型原始语义的参照系。

`torch2vk` 不是要抹掉这个参照系，而是要始终保留它。  
系统中的每一个重要 shader variant，都应该能放回 PyTorch reference 的语义框架中理解。

### 2. reference 执行源码是直接、可读的模型翻译

reference 执行源码的意义首先不是快，而是正确、可见、可解释。

它应该尽量贴近模型原始结构，让人和 AI 能够在 PyTorch 模块结构和 `torch2vk` 的模型侧执行源码之间机械地来回映射。

reference 执行源码的价值在于：

1. 作为正确语义的基线。
2. 作为之后所有优化版本的对照对象。
3. 作为调试和诊断时的解释基准。

### 3. optimized 执行源码是显式改写后的新版本

优化后的执行源码不应该是“reference 版本加上一堆不可见 pass 的结果”。

在 `torch2vk` 里，optimized 执行源码应该是系统中另一个显式存在的版本。  
它的存在方式应该仍然是源码，而不是某个隐藏流程的副产物。

这意味着优化逻辑本身也是系统对外可见的一部分，而不是内部黑盒。

### 4. shader 必须匹配 checkpoint 权重

`torch2vk` 的 safetensors port 不通过权重转换来适配 shader。

权重来自 checkpoint 之后，应保持声明的 dtype、shape 和 layout。  
如果某个 shader variant 需要另一种权重布局、打包形式或量化形式，那么这个 variant 不适合当前 port。

系统应该始终能回答：

1. shader 消费的是否就是原始 safetensors 权重表达。
2. shader 的 binding、dtype、shape、layout 是否和权重声明一致。
3. 如果不一致，应该换 shader，而不是在 runtime 中悄悄转换权重。

### 5. agent 是执行优化者

`torch2vk` 的一个核心前提是：优化者可以是 AI，而且 AI 需要直接理解执行源码和方法本身。

因此系统最终应该允许 agent 做的，不只是“调用接口跑一下”，而是：

1. 读懂一个 shader variant。
2. 改写一个 shader variant。
3. 理解某次优化是否改变了语义边界。
4. 找到 reference 和优化版本第一次分歧的位置。

这也是为什么系统必须偏向源码可见性，而不是内部抽象层的完整性。

## 性能观

`torch2vk` 的性能观不应该建立在“尽量把一切都藏进 runtime”上，而应该建立在几个明确前提上。

### 1. Python 负责组合，Vulkan 负责计算

Python 是控制面。  
Vulkan 是数据面。

Python 负责：

1. shader variant 的表达。
2. shader 的选择。
3. 多步执行在 Python 源码中的组合。
4. 比较、诊断和控制流程。

Vulkan 负责：

1. 重计算。
2. 权重相关 kernel。
3. 必须留在 device 上的状态更新。
4. 真正的执行成本。

### 2. device residency 是系统契约

这个系统的性能观应该建立在 device residency 上。  
如果多步流程之间的张量和状态因为 API 设计被迫回到 host，那系统边界就是错的。

device residency 在 `torch2vk` 里不只是一个优化建议，而应该是：

1. shader 输入输出的契约。
2. 持久状态存在方式的契约。
3. 跨 shader handoff 的契约。
4. 性能判断的基本前提。

### 3. repeated shader sequence 应该有稳定执行形态

对于会重复出现的 shader sequence，系统最终应当具备“稳定执行形态”的概念。

这里重要的不是某种具体机制名称，而是：

1. 重复调用的 shader sequence 不应每次都像第一次一样重新构造全部执行语义。
2. 稳定 workload 下的执行形态应该是可重用的。
3. shader 的输入输出边界、状态边界和资源边界应当足够稳定，才能支持真正高效的重复执行。

`torch2vk` 不需要因为这个目标变成通用 command compiler，但它的最终形态必须承认：  
重复 shader sequence 的性能，不只是 kernel 本身，还和 shader 级别的稳定执行成本有关。

当 CPU 侧反复驱动 eager dispatch 成为瓶颈时，系统应该进一步支持 Vulkan 录制与回放。

这里它更接近 CUDA Graph 一类机制：

1. 录制的是一段已经稳定下来的 Vulkan 提交序列。
2. 回放是为了减少 CPU 驱动和提交开销。
3. 它服务的是已有 shader 或 shader 组合的执行效率。
4. 它不是新的执行语义来源，也不是新的 workflow 抽象层。

## 它想提供什么体验

`torch2vk` 最终想提供的不是“隐藏复杂度”，而是“让复杂度处在正确的位置上”。

对人和 AI 来说，理想体验应该是：

1. 想理解一个执行片段时，直接读对应 shader 和模型侧 Python 代码。
2. 想看权重语义时，直接看对应模型家族的权重接口。
3. 想知道状态怎么流动时，直接看 shader 输入输出和 Python 源码里的组合方式。
4. 想定位数值偏差时，能快速知道第一处偏差出现在哪里。
5. 想做一个优化版本时，能把新旧版本并排理解，而不是猜 runtime 发生了什么。

这个系统要降低的，不是“写代码的劳动量”本身，而是：

1. 理解系统时的不透明成本。
2. 修改系统时的心理负担。
3. 调试系统时的定位成本。
4. 优化系统时的不可控风险。

## 它最重要的几个性质

### 1. 可读

重要执行逻辑必须能直接阅读和讨论，而不是只能间接推测。

### 2. 可验证

系统必须始终保留“我能不能知道它和 reference 在哪里开始不同”这个能力。

### 3. 可改写

优化不应建立在“不要碰它”的脆弱结构上，而应允许人和 AI 明确改写执行源码和 shader。

### 4. 可组合

系统不应只擅长单次 forward，还应支持在 Python 源码中把多个 shader 调用直接组合成真实多步执行。

### 5. 可比较

不同 shader variant、不同布局、不同优化形式之间，应该能够清楚比较，而不是只能二选一替换。

### 6. 可诊断

系统应该偏向让数值偏差、布局偏差、权重偏差、kernel 偏差更容易被定位，而不是更难被解释。

## 它明确不是什么

`torch2vk` 不应该变成下面这些东西：

1. 通用 PyTorch 执行器。
2. 通用 compiler framework。
3. 多 backend 抽象层。
4. 依赖导出图、IR 或隐藏 lowering 才能理解的系统。
5. 默认通过 CPU 往返来串联多步执行的系统。
6. 以“统一抽象”名义抹掉模型家族差异的系统。
7. 以“自动优化”名义让真实执行过程变得不可解释的系统。
8. 把重要执行语义埋进 runtime 内部对象模型的系统。

这些方向的问题不在于它们本身毫无价值，而在于它们会把 `torch2vk` 带去另一个目标：  
从“可见、可控、可比较的 shader-first 系统”变成“抽象层越来越厚的执行平台”。

## 为什么是这种形态

因为这个项目真正关心的，不只是能跑。

它更关心：

1. 能不能知道自己在跑什么。
2. 能不能知道自己为什么对。
3. 能不能知道自己为什么错。
4. 能不能在不把系统变黑盒的前提下继续优化。
5. 能不能让 AI 真正参与阅读、修改和迭代，而不是只会调用外层接口。

如果一个系统为了省去部分显式表达，最终换来的是更难理解、更难验证、更难优化，那它就不符合 `torch2vk` 的目标。
