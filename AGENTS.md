# 保持代码简洁实用

* 兼容和fallback会导致不必要的 if else，让代码变得更复杂
* 用 object, any 会导致类型字段不清晰，且无法用 pyright 静态保证引用字段存在
* 运行时的状态越多，需要维护状态一致性就需要越多的代码。一个single source of truth，减少 mutable state 可以减少代码量
* 函数的一部分字段只给一部分call site使用，说明了过度复用的 code smell。这会导致阅读每个call site 的代码，都需要关心其他地方的业务逻辑。不应该通过加字段来强行复用。
* 能够直接从a转换到b的，就不要中间增加c的概念，从a到c,然后从c到b。应该整体思考目标，避免不必要的中间过程。
* 异常处理应该在运行的入口处，如果不确定异常在哪里处理应该找到运行入口。然后在最外层的入口处添加 try catch。不要在内部做 try catch。
* 任何时候不应该catch之后静默吞掉异常
* import 始终在文件头部，而不是在函数内部
* 不要使用单元测试，使用尽可能大范围的集成测试替代。集成测试的边界应该是相对来说业务逻辑变更不太频繁的稳定边界。单元测试的被测接口因为会经常被修改，所以测试价值不大。
